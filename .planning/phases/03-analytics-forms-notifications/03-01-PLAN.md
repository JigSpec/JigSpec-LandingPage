<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → Vercel env-var store | User pastes the project API key into Vercel dashboard / CLI. Trusted endpoint (Vercel HTTPS). |
| User input → local `.env.local` | User pastes the same key into a gitignored local file. Trust boundary is the local filesystem. |
| Local repo → public GitHub remote | The `.gitignore` patch is the boundary that prevents the key from crossing into a public artifact. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Severity | Mitigation Plan |
|-----------|----------|-----------|-------------|----------|-----------------|
| T-03-01-01 | Information disclosure | `.env.local` accidentally committed to public GitHub repo | mitigate | High | Task 3 patches `.gitignore` to explicitly ignore `.env.local`; Task 4 verifies via `git check-ignore -v` before any commit. Defense-in-depth: PostHog project API keys are publishable per PostHog docs (they only allow event ingestion, not data extraction), so even an accidental leak is recoverable by rotating the key. |
| T-03-01-02 | Information disclosure | API key pasted into chat / commit message / `.env.example` | mitigate | Medium | Task 1 acceptance only requires the user to share the key prefix (first 5 chars). `.env.example` per Task 3 has an empty value after `=`. Plan SUMMARY must not echo the key. |
| T-03-01-03 | Tampering | Wrong env-var name (`POSTHOG_KEY` without `PUBLIC_` prefix) silently breaks the deploy | mitigate | High | Task 2 acceptance criteria + Pitfall 2 explicitly name the literal `PUBLIC_POSTHOG_KEY`. Plan 03-03's task asserts that `import.meta.env.PUBLIC_POSTHOG_KEY` is non-undefined at build time. |
| T-03-01-04 | Denial of service | Vercel env var set on Production only → preview canary fails ANALYTICS-05 | mitigate | High | Task 2 acceptance requires `vercel env ls` output showing all three scopes (Pitfall 9 — explicit three-scope verification). |
| T-03-01-05 | Elevation of privilege | User accidentally captures the `personal API key` instead of the project API key | accept | Low | Task 1 instructions specify "project API key only" and reject the personal key explicitly. PostHog UI surfaces these in different sections; user mistake is unlikely. |
</threat_model>
